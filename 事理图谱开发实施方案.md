## 1. 总体开发策略

本功能不要直接在主系统里边想边改。先在当前目录形成“可跑、可测、可解释”的轻量原型，再把稳定接口迁移到主系统。

采用三层推进：

1. 原型层：当前目录 `event_graph_dev/`，只读读取主系统数据库和项目 JSON，完成教师思路解析、图谱定位、插入位评分和项目草案结构。
2. 集成层：把原型中稳定的模型、服务、测试迁回主系统 `backend/app/` 和 `backend/tests/`。
3. 部署层：主系统代码推送到远程后，由服务器拉取指定提交或 tag，完成迁移、重启和健康检查。

这样做的原因：

- 当前目录可以快速实验，不污染主系统。
- 主系统集成时接口已经稳定，风险更小。
- 服务器更新只基于已推送的 Git 提交，便于回滚。

### 任务 6：服务器更新

服务器更新不能从当前目录直接更新。必须先完成主系统回接、测试、commit 和 push。

上线前检查：

- 远程分支存在。
- CI 或本地测试通过。
- 数据库已备份。
- 已确认部署路径和服务名。
- 已准备回滚 tag 或上一稳定 commit。

部署顺序：

```bash
git fetch --all --tags
git checkout <release-branch-or-tag>
python3 -m pytest backend/tests/test_event_graph_planner.py -v
systemctl restart <backend-service>
curl -f <server-health-url>
```

具体命令要根据服务器部署方式补全。没有服务器地址、服务名和部署目录时，不执行更新服务器。

## 4. Git 推送规则

用户要求“每一步推送 git”，执行规则如下：

1. 每个任务完成一个可验证小步。
2. 先跑测试。
3. 再 commit。
4. 立刻 push。
5. push 失败时停止并记录原因。

当前状态与阻塞：

- 当前目录已初始化为本地 Git 仓库。
- 当前目录已有首次提交：`b164adb chore: initialize event graph workspace`。
- 主系统目录不是 Git 仓库。
- 当前未配置远程地址。

因此本次可以完成文件和代码落地，但不能声称已完成 `git push`。要满足每步 push，需要先提供或创建远程仓库地址。
