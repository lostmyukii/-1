## 1. 总体开发策略

本功能默认不直接在主系统里边想边改。先在当前目录形成“可跑、可测、可解释”的轻量原型，再把稳定接口迁移到主系统。用户明确要求在 `/Users/yukii/Desktop/Kimi_Agent_AI Scratch功能升级` 落代码时，按功能分支和增量文件策略执行。

采用三层推进：

1. 原型层：当前目录 `event_graph_dev/`，只读读取主系统数据库和项目 JSON，完成教师思路解析、图谱定位、插入位评分和项目草案结构。
2. 集成层：把原型中稳定的模型、服务、测试迁回主系统 `backend/app/` 和 `backend/tests/`。本轮采用增量回接，不提交历史文档、数据库备份、视频产物、本地依赖和缓存。
3. 部署层：主系统代码推送到远程后，由服务器拉取指定提交或 tag，完成迁移、重启和健康检查。

这样做的原因：

- 当前目录可以快速实验，不污染主系统。
- 主系统集成时接口已经稳定，风险更小。
- 服务器更新只基于已推送的 Git 提交，便于回滚。

新增 `generated_project_drafts` 独立表而不是复用 `graph_edges.edge_metadata` 的原因：草案记录具有独立生命周期、导出文件血缘、审核摘要和后续回滚线索；这些不是图谱边关系的轻量 metadata，必须能被 API 独立查询和审计。当前切片仍只保存 `draft`，不写入 `project_packages`，不触发 RAG，不绕过 ProjectPackage validate/review/approve/activate。

新增 `project_graph_review_records` 记录草案审核意见和候选图谱变更，但仍不修改 `ProjectPackage` 状态。审核记录的 `ready_for_project_package_review` 只表示“可进入项目包校验/提审”，不是 `active`。

新增 ProjectPackage 校验/提审衔接服务复用现有 `project_plugin_service.read_package_payloads`、`upsert_package_from_manifest`、`validate_project_package` 和 `set_package_status`，只在教研审核记录已标记 `ready_for_project_package_review` 时导入草案目录。导入时强制保持 `draft`、`rag_indexable=false`、`recommendable=false`，校验无错误后最多流转到 `in_review`，不允许直接 `active`，因此不会触发 RAG 索引。

新增 ProjectPackage 提审权限边界：`project-package-review` 接口复用主系统 `require_admin` 管理员权限，未登录请求返回 401，教师账号返回 403，提审 actor 以认证用户 ID 为准，不接受请求体冒用。
