本方案要解决的问题不是“再做一个知识点推荐器”，而是建立一个能回答“为什么这个思路应该放在这里”的系统：

- 老师输入一句话思路后，系统能判断它应该插入哪个 L1-L5 阶段、哪个项目类型、哪个项目前后。
- 系统能解释该思路前置需要哪些知识点，后续会通向哪些知识点或项目。
- 系统能判断是复用现有项目、生成变体项目，还是补一个新的可拔插项目。
- 系统能直接生成项目插件包草案，经过 schema、知识点、年龄难度、AI 边界、七维评价、教研审核后进入知识库。

## 2. 设计原则

### 2.1 事理一体

普通知识图谱偏“点和边”，事理图谱需要同时描述“发生了什么”和“为什么这样安排”。

- “事”：教师想法、项目任务、课堂活动、学生证据、审核记录、项目入库事件。
- “理”：知识点先后、认知阶段、难度约束、硬件条件、AI 使用边界、项目推荐理由、审核通过理由。

### 2.2 先定位，再生成

一句话备课不能直接生成项目。必须先做图谱定位：

1. 识别教师思路的主题、能力、交付物、硬件、软件和风险。
2. 匹配年龄层、知识点、能力点和已有项目。
3. 找到最合理的插入位。
4. 决定复用、改造、生成新项目。
5. 生成项目包草案并进入审核。

### 2.3 可拔插，不污染主库

所有新项目先是 `draft` 项目包，不直接进入活跃项目库。只有通过校验和教研审核后，才变为 `active` 并触发 RAG 索引和图谱边更新。

可插入意味着能接到现有知识点和项目序列中；可拔出意味着能 `deprecated`、`archived`、`replaced` 或 rollback，并保留审计记录。

### 2.4 年龄与认知负荷优先

生成项目必须先服从年龄段和认知负荷，再谈创意：

| 阶段 | 年龄 | 生成边界 |
| --- | --- | --- |
| L1 | 3-6 | 教师端辅助，AI 不直接面向孩子，不进入正式竞赛推荐 |
| L2 | 7-9 | AI 可生成结构脚手架，辅助比例不超过 50% |
| L3 | 10-12 | 可生成模块化方案和数据/AI 项目框架，辅助比例不超过 30% |
| L4 | 13-15 | 强调算法、系统、竞赛策略，辅助比例不超过 20% |
| L5 | 16-18 | 强调架构、科研、产品化和伦理，辅助比例不超过 10% |

## 4. 事理图谱核心对象

### 4.1 节点类型

| 节点 | ID 前缀建议 | 说明 |
| --- | --- | --- |
| 教师思路 | `IDEA-` | 一句话备课输入及解析结果 |
| 插入位 | `SLOT-` | 某个思路在项目序列中的推荐位置 |
| 项目草案 | `DRAFT-` | 生成中的项目插件草案 |
| 项目包 | `PKG-` | 对应 `ProjectPackage.package_id` |
| 项目 | 现有 `project_id` | 已入库或可推荐项目 |
| 项目组 | `KP-SET-` | 一组变体项目 |
| 知识点 | `KP-` | 已有知识点 |
| 能力点 | `COMP-` | 已有能力节点 |
| 竞赛/考试点 | `CMP-` | 已有竞赛点或考试映射 |
| 年龄阶段 | `AGE-L1` 等 | L1-L5 |
| 硬件 | `HW-` | 套件、传感器、主控、设备 |
| 交付物 | `DLV-` | 作品、报告、演示、数据集、答辩材料 |
| 审核事件 | `REVIEW-` | 教师、教研、系统校验和入库记录 |
| 课节 | `LESSON-` | 生成项目中的课节节点 |

现有 `graph_node` 对 `KP-`、`CMP-`、项目、能力、`AGE-`、`HW-`、`DLV-` 已有基本支持。后续实现时只需扩展 `IDEA-`、`SLOT-`、`DRAFT-`、`PKG-`、`REVIEW-`、`LESSON-` 的展示逻辑。

实现提醒：`KP-SET-` 以 `KP-` 开头，扩展 `node_type_for_id` 时要先判断 `KP-SET-`，再判断普通 `KP-`，避免项目组被误判为知识点。

复用现有关系：

- `supports_project`
- `belongs_to_competency`
- `suitable_for_age_band`
- `applied_in_competition`
- `requires_hardware`
- `produces_deliverable`
- `recommended_next`
- `successor_of`
- `prerequisite_of`

| 关系 | 起点 -> 终点 | 用途 |
| --- | --- | --- |
| `idea_targets_knowledge` | IDEA -> KP | 教师思路命中的知识点 |
| `idea_fits_stage` | IDEA -> AGE | 思路适合的年龄阶段 |
| `idea_matches_project` | IDEA -> Project | 思路可复用的现有项目 |

## 5. 一句话备课流程

### 5.1 输入

教师可以输入一句话，也可以带少量约束：

```json
{
  "idea_text": "我想让孩子做一个用语音控制小车完成送货任务的项目",
  "class_id": "optional",
  "age": 8,
  "age_band": "L2",
  "current_project_id": "optional",
  "known_knowledge_point_ids": [],
  "available_hardware": ["mTiny", "Arduino", "超声波传感器"],
  "duration_hours": 10,
  "target_use": "classroom",
  "teacher_constraints": ["希望有展示环节", "不要太偏竞赛"]
}
```

### 5.2 解析

解析层输出结构化标签：

- 主题：语音控制、小车、送货、任务执行。
- 场景：物流、校园、生活服务。
- 能力：事件触发、条件判断、序列规划、传感器或语音输入、展示表达。
- 硬件：小车、语音模块、主控、传感器。
- 交付物：能运行的送货小车、任务路线图、演示说明。
- 风险：L1 不可让孩子直接使用 AI 生成；L2 不能生成完整代码；硬件复杂度需控制。

### 5.3 图谱定位

系统按顺序判断：

1. 年龄阶段：由年龄、班级、当前项目推断 L1-L5。
2. 认知阶段：是否属于前置体验、同步承载、延伸挑战、复盘修复、竞赛包装或综合评价。
3. 知识点命中：匹配 `knowledge_points` 和 `GraphEdge.prerequisite_of`。
4. 现有项目命中：用 `project_recommendation_engine` 的关键词、知识点和项目元数据做初筛。
5. 插入位：结合 `recommended_next`、`successor_of`、`prerequisite_of`、同一 ProjectSet 变体和当前课程进度。

输出示例：

```json
{
  "stage_decision": {
    "age_band": "L2",
    "slot_type": "synchronous_carrier",
    "confidence": 0.84,
    "reason": "8岁学生适合在具体运算阶段用小车任务承载事件、序列、条件和电机控制。"
  },
  "before_knowledge_points": [
    "C2-L2-02 事件驱动",
    "C1-L1-04 序列指令",
    "M2-L2-02 角度与方向"
  ],
  "core_knowledge_points": [
    "C2-L2-03 条件判断",
    "P2-L2-06 电机控制基础",
    "E2-L2-01 项目规划"
  ],
  "after_knowledge_points": [
    "P2-L2-05 超声波测距",
    "E2-L2-03 测试调试",
    "C3-L3-02 函数与模块"
  ],
  "candidate_existing_projects": [
    "WSL-ARD-SH-L2-007A Arduino超声波避障小车",
    "WSL-ARD-SH-L2-009B 快递分拣机器人",
    "WSL-MIX-SH-L2-006B 智能自动窗帘系统"
  ]
}
```

## 6. 项目插入策略

系统不只回答“推荐哪个项目”，还要回答“插在何处”。

| 插入类型 | 触发条件 | 处理方式 |
| --- | --- | --- |
| 前置补齐 | 教师思路依赖学生未掌握知识点 | 生成短项目或任务卡，插在目标项目之前 |
| 同步承载 | 思路刚好覆盖当前阶段核心知识 | 复用或生成同层项目，作为主项目 |
| 延伸挑战 | 思路超出当前项目，但仍在可承受范围 | 生成 B/C 变体或挑战模块，插在项目之后 |
| 复盘修复 | 学生证据显示某知识点薄弱 | 生成修复性小项目，关联失败证据 |
| 竞赛包装 | 项目已成熟，目标是竞赛或综评 | 生成竞赛映射、材料清单、答辩和证据包 |
| 综合评价 | 需要检验多个知识点迁移能力 | 生成跨知识点综合项目 |

插入位计算建议：

```text
slot_score =
  0.25 * stage_fit
  + 0.20 * knowledge_overlap
  + 0.15 * prerequisite_readiness
  + 0.15 * project_gap_value
  + 0.10 * hardware_fit
  + 0.10 * deliverable_fit
  + 0.05 * teacher_goal_fit
  - risk_penalty
```

关键风险扣分：

- 年龄段不匹配。
- 难度超过 `AGE_RULES` 上限。
- L1 直接进入竞赛或让孩子直接使用生成式 AI。
- 缺少前置知识点。
- 硬件不可用。
- 生成项目与现有项目高度重复。
- AI 输出超出防认知卸载边界。

## 7. 项目生成机制

### 7.1 生成决策

图谱定位后有三种生成路径：

| 路径 | 条件 | 输出 |
| --- | --- | --- |
| 复用现有项目 | 已有项目匹配度高，知识点和硬件均合适 | 返回项目推荐、前后知识点、实施建议 |
| 生成项目变体 | 有相近项目，但场景、硬件或目标用户不同 | 生成同一 ProjectSet 下的新 variant |
| 生成新项目包 | 图谱发现明显知识缺口或课程新方向 | 生成新的 ProjectPackage 草案 |

### 7.2 项目包输出

生成的新项目必须落成项目插件包，不允许只生成一段描述。项目包包含：

- `project.json`：项目元数据、状态、年龄、难度、知识点、能力点、硬件软件、可推荐状态。
- `curriculum.md`：课程目标、项目背景、关键活动、教师实施说明。
- `lesson_plan.json`：课节计划，每节绑定知识点、证据、评价和风险。
- `knowledge_map.json`：核心知识点、AI 能力点、前置和后续知识点。
- `materials.json`：硬件、软件、耗材、替代方案、安全检查。
- `assessment.json`：七维评价体系。
- `ai_guardrails.json`：AI 使用边界、防认知卸载、输出校验。
- `zongping_map.json`：综评材料映射。
- `competition_map.json`：考试/竞赛映射。
- `parent_view.md`：家长可读说明。
- `teacher_notes.md`：教师实施、常见失败和追问话术。
- `README.md`：项目包说明。

### 7.3 生成边界

对“一句话项目设计”和低龄项目，系统遵守以下规则：

- 可以生成角色、任务、材料、知识点、项目结构、评价量表和教师提示。
- 不直接生成低龄学生可复制粘贴的完整代码。
- Scratch 项目只生成骨架，不生成可运行积木链。
- 对 Python/C++ 高阶项目，可以生成模块接口、伪代码和测试要求，但必须保留学生设计、实现、解释和调试任务。
- 每个生成项目必须写明“学生必须完成的任务”和“AI 不能代劳的任务”。

## 8. 审核与入库流程

### 8.1 状态流转

```text
teacher_idea
  -> graph_positioned
  -> draft_project_package
  -> validate
  -> in_review
  -> approved
  -> active
  -> rag_indexed
  -> graph_edges_materialized
```

现有系统已经支持项目包流转：

- 上传或生成项目包后为 `draft`。
- 校验通过后进入 `in_review`。
- 教研审核通过后为 `approved`。
- 激活后为 `active`，项目变为可推荐，并触发 RAG 索引。

### 8.2 自动校验

自动校验至少包括：

| 校验 | 通过标准 |
| --- | --- |
| schema | 必填字段和必需文件齐全 |
| knowledge_map | 知识点和 AI 能力点 ID 存在 |
| alignment | L2+ 有考级/竞赛映射且不是纯规则兜底 |
| age_fit | 难度不超过年龄段上限 |
| ai_guardrail | 有 AI 能力、学生必做任务、AI 输出校验、防认知卸载和中文边界说明 |
| assessment | 七维评价完整：问题定义、逻辑建模、AI协作、代码/硬件实现、数据与证据、迭代改进、交付表达 |
| duplication | 与现有项目不是低价值重复 |
| graph_consistency | 前置、后续、插入位和项目知识点不矛盾 |
| lesson_quality | 课节有目标、活动、证据、评价、失败排查 |

### 8.3 教研审核

教研审核不只看文案，还要看图谱解释：

- 为什么适合该年龄段。
- 为什么插在这个项目之前或之后。
- 前置知识点是否足够。
- 后续知识点是否自然。
- 是否过度依赖 AI。
- 是否有真实交付物。
- 是否能被教师在现有设备条件下实施。
- 是否应该作为独立项目、变体项目、挑战模块或复盘模块。

## 9. 知识库与 RAG 入库

项目激活后，系统生成 RAG chunks：

- 家长解释 chunk。
- 教师实施 chunk。
- 知识点映射 chunk。
- 项目推荐 chunk。
- 考级竞赛映射 chunk。

事理图谱还应额外生成：

- 插入理由 chunk：解释项目为什么在这个阶段出现。
- 前后知识链 chunk：列出前置、核心、后续知识点。
- 审核结论 chunk：保存教研审核意见和修改记录。

这些内容进入 RAG 后，教师后续问“为什么推荐这个项目”“这个想法适合哪节课”“我能不能换成另一个场景”时，系统能用可追溯材料回答。

## 12. 推荐与生成算法

### 12.1 检索层

检索来源：

- `projects`：项目元数据、项目标签、硬件软件、知识点、教案、AI 边界。
- `knowledge_points`：知识点名称、描述、年龄段、前置知识。
- `graph_edges`：前后关系、项目关系、硬件和交付物关系。
- `project_packages`：项目生命周期、版本、审核状态。
- `project_rag_index_jobs`：可检索 chunk。

### 12.2 排序层

排序特征：

- 年龄段适配。
- 主题关键词命中。
- 知识点重合。
- 前置知识是否满足。
- 与当前项目的 `recommended_next` 或 `successor_of` 关系。
- 设备可用性。
- 交付物匹配。
- 目标用途匹配：课堂、家长展示、竞赛、综评、作品集。
- 项目重复度。
- AI 使用风险。

### 12.3 生成层

生成顺序：

1. 生成项目一句话定位。
2. 生成前置、核心、后续知识链。
3. 生成项目结构：目标用户、真实问题、核心任务、交付物。
4. 生成课节计划。
5. 生成材料和安全检查。
6. 生成七维评价。
7. 生成 AI guardrails。
8. 生成家长视角和教师说明。
9. 生成项目包文件。
10. 运行自动校验。

## 13. 示例：教师一句话到项目草案

教师输入：

> 我想让孩子做一个用语音控制小车给同学送物资的项目。

图谱判断：

- 如果学生为 L1：更适合作为“语音指令小车”的感知体验，不生成孩子端项目，只生成教师演示与具身活动。
- 如果学生为 L2：适合作为“事件驱动 + 序列规划 + 电机控制”的软硬结合项目，可插在“Scratch 坐标迷宫探险家”之后、“Arduino 超声波避障小车”之前。
- 如果学生为 L3：可以升级为“Python 串口 + Arduino 小车 + 任务状态机”的系统项目，可插在“智能星球车探测系统”前作为轻量过渡。

L2 项目草案：

- 名称：语音送货任务车。
- 插入类型：同步承载 + 延伸挑战。
- 前置知识点：事件驱动、序列指令、角度与方向。
- 核心知识点：条件判断、电机控制、项目规划、展示表达。
- 后续知识点：超声波测距、测试调试、状态机入门。
- 项目产出：一辆可完成 2-3 个送货任务点的小车、路线图、测试记录、展示视频。
- AI 边界：AI 可识别语音命令或辅助生成任务卡，学生必须自己规划路线、接线、调参、解释失败原因。
- 插入位：`plugs_after -> Scratch坐标迷宫探险家`，`plugs_before -> Arduino超声波避障小车`。

## 15. 分阶段实施计划

### 阶段一：图谱定位 MVP

目标：教师输入一句话，系统返回阶段、知识点链、候选项目和插入位。

范围：

- 新增 `teacher_ideas` 和 `project_insertion_slots`。
- 扩展 `graph_node` 支持 `IDEA-` 和 `SLOT-`。
- 复用 `project_recommendation_engine` 做候选项目召回。
- 输出可解释 JSON，不生成项目包。

验收：

- 对 20 条教师思路样例，能稳定返回 L1-L5 阶段和至少 3 个可解释证据。
- L1 不误推竞赛。
- 能明确列出 before/core/after 知识点。

### 阶段二：项目草案生成

目标：对“需要生成”的思路生成 ProjectPackage 草案。

范围：

- 新增 `generated_project_drafts`。
- 生成 12 个项目包文件。
- 调用现有项目包 validate。
- 生成图谱边：`idea_targets_knowledge`、`plugs_before`、`plugs_after`、`generated_package`。

验收：

- 生成项目包能通过 schema、knowledge_map、age_fit、ai_guardrail、assessment 校验。
- 生成结果默认 `draft`，不能绕过审核激活。
- 项目重复度检查能拦截低价值重复项目。

### 阶段三：教研审核与入库

目标：让审核通过的项目进入 active 项目库、RAG 和图谱。

范围：

- 新增 `project_graph_review_records`。
- 审核台展示图谱理由与校验结果。
- 复用 approve、activate、rag-index 流程。
- 激活后物化图谱边。

验收：

- 审核记录可追溯。
- active 后项目可被推荐。
- RAG 能回答“为什么这个项目插在这里”。

### 阶段四：课堂闭环

目标：把学生学习证据接回图谱，驱动下一次备课。

范围：

- 接入课堂 observation、failed-items、student_knowledge_progress。
- 对薄弱知识点自动建议复盘项目。
- 对高掌握学生建议延伸挑战或竞赛包装。

验收：

- 能基于学生证据生成复盘插入位。
- 能把项目完成证据映射到知识点掌握状态。

## 16. 风险与控制

| 风险 | 控制 |
| --- | --- |
| 教师一句话过短导致误判 | 返回置信度和缺失信息，不确定时只推荐不生成 |
| 新项目重复污染库 | 做项目名称、知识点、硬件、交付物和 RAG 相似度去重 |
| 生成项目过难 | 强制 `age_fit` 和难度上限校验 |
| AI 代替学生思考 | `ai_guardrails.json` 必填，且校验学生必做任务 |
| 项目无法实施 | materials 和 safety 必填，审核时检查设备可用性 |
| 图谱边过多变脏 | 低置信度边先标记为候选，只在审核通过后物化为正式边 |
| 入库不可追溯 | 保留 ProjectAuditLog、review record、RAG chunk 和 graph edge metadata |

## 17. 成功指标

产品指标：

- 教师一句话定位成功率 ≥ 85%。
- 教师接受“复用/变体/新项目”建议比例 ≥ 70%。
- 新项目包一次通过自动校验比例 ≥ 80%。
- 教研审核退回率逐月下降。

教学指标：

- 每个新项目至少绑定 5 个有效知识点或能力点。
- 每个项目有明确前置和后续知识点。
- 低龄项目无直接代码代劳和竞赛误推。
- 项目完成后有可观察证据进入知识库。

系统指标：

- 项目入库全流程可审计。
- RAG 能引用项目包、知识点、审核结论回答推荐理由。
- 图谱边新增后不破坏现有 coverage 统计。

## 18. 结论

事理图谱的核心价值，是把当前系统已经存在的 100+ 项目、知识点、竞赛映射和项目插件机制串成一个能解释、能生成、能审核的教学决策网络。

对教师来说，它不是多一个后台配置项，而是把“一句话想法”翻译成“这个孩子现在最适合做什么项目，为什么，前面要补什么，后面能通向哪里”。

对系统来说，它不是另起炉灶，而是在现有 `projects`、`knowledge_points`、`graph_edges`、`project_packages`、RAG 索引和审核流之上补上“思路进入系统”的入口和“项目生成入库”的闭环。
